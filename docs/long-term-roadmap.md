### Phase 2：数据持久化升级（Week 5-6）

> 目标：从 JSON 文件走向可靠存储，支持多设备同步

**方案选择**（二选一）：

| 方案 | SQLite + Better-SQLite3 | PostgreSQL |
|------|------------------------|------------|
| 优点 | 零运维、单文件、适合单机 | 多用户、强一致性、成熟生态 |
| 缺点 | 不支持多用户写入 | 需要运维数据库服务 |
| 适用 | 个人/小团队 | 团队协作 |

**推荐**：先 SQLite，后续按需迁移到 PostgreSQL

**P2.1 数据库层**
- [ ] 设计 schema（projects, tasks, milestones, phases, files, activities, ai_notes）
- [ ] 实现 Drizzle ORM 或 Prisma 迁移
- [ ] 迁移现有 JSON 数据到 SQLite
- [ ] 保留 JSON 导入/导出能力（兼容性）

**P2.2 API 重构**
- [ ] REST → 更完整的 CRUD 端点
- [ ] 添加分页支持（项目列表、活动历史）
- [ ] 添加过滤/搜索 API
- [ ] WebSocket 替代 SSE（支持双向通信）

**P2.3 数据备份**
- [ ] 自动备份机制（每日导出 JSON）
- [ ] 版本快照（支持回滚到任意时间点）
- [ ] 导出为 MS Project XML / Excel

## 五、关键决策点

### 决策 1：SQLite vs PostgreSQL
- **触发条件**：Phase 2 开始前
- **决策依据**：用户规模是否 > 5 人同时在线
- **建议**：先 SQLite，遇到瓶颈再迁移

### 决策 2：实时协作方案
- **触发条件**：Phase 3 开始前
- **选项**：Yjs (CRDT) vs Liveblocks vs 自研 OT
- **建议**：Yjs，成熟度高、社区活跃、与 React 集成好

### 决策 3：移动端方案
- **触发条件**：Phase 5 开始前
- **选项**：PWA vs React Native vs Expo
- **建议**：PWA，与现有技术栈一致、维护成本最低

### 决策 4：AI 调用策略
- **触发条件**：Phase 1 开始前
- **选项**：纯前端调用 vs 后端代理
- **建议**：后端代理，避免 API Key 暴露、可做成本控制
